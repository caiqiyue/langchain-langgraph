## <center>第3阶段、图状态中处理消息的思路</center>

### 1. 消息处理概述

&emsp;&emsp;Reducer机制的一个现实意义是：我们可以基于这种方式去构建历史对话记录。因为目前大多数大模型应用都是接受消息列表作为输入，就像LangChain中的Chat Model，需要接收Message对象列表作为输入。这些消息有多种形式，例如HumanMessage（用户输入）或AIMessage（大模型响应）。

### 2. 大模型接入与消息处理

&emsp;&emsp;下面这个示例展示了如何进一步将大模型接入到LangGraph工作流程中，并允许动态消息处理以及与模型的交互。

&emsp;&emsp;**代码结构**：
```python
import operator
from typing import Annotated, TypedDict, List

class State(TypedDict):
    messages: Annotated[List[str], operator.add]
```

&emsp;&emsp;在这里，第一个节点调用大模型并生成一个输出，该输出是一个AIMessage对象类型。然后，第二个节点直接将前一个节点的AIMessage提取为具体的JSON格式。

### 3. 消息流程图

&emsp;&emsp;当定义状态模式的结构发生变化以后，在节点函数中的读取和存储逻辑也要发生相应的变化：

1. 初始状态包含用户输入消息
2. chat_with_model节点接收消息列表，调用大模型
3. 返回的AIMessage通过operator.add追加到messages列表
4. convert_messages节点处理消息列表，进行格式转换

### 4. 关键概念

&emsp;&emsp;**operator.add的作用**：

&emsp;&emsp;在消息处理场景中，简单地将消息追加到现有列表中，就是通过operator.add实现的。这种方式确保了：
- 新消息被追加到现有消息列表
- 保留完整的历史对话记录
- 每个节点都可以访问完整的消息历史

### 5. 拓展思考

#### 5.1 深度概念

&emsp;&emsp;**为什么选择operator.add而不是覆盖？**

&emsp;&emsp;在对话系统中，保留完整的对话历史至关重要。通过使用operator.add作为Reducer，我们可以：
- 追踪完整的对话上下文
- 让大模型看到之前的对话内容
- 支持多轮对话场景

#### 5.2 实践建议

&emsp;&emsp;在现实开发情况下，将先前的对话历史记录存储为图形状态中的消息列表是必须要做的。实现这一功能，在LangGraph框架中可以向存储Message对象列表的图状态添加一个键（通道），并使用Reducer函数对其进行注释，从而告诉图在每次状态更新时（例如，当节点发送更新时）更新状态中的Message对象列表。
