## <center>第五阶段、自定义中间件应用</center>

### 1. 中间件的参数传递机制

#### 1.1 ModelRequest/Response

单次调用级别的"细粒度控制"，作用域仅限于单次模型或工具调用的生命周期。

| **字段** | **说明** | **可修改性** |
|---------|---------|------------|
| model | 当前要调用的模型实例 | 可修改 |
| messages | 消息列表 | 可修改 |
| tools | 可用工具列表 | 可修改 |
| state | Agent 的当前状态 | 只读 |
| runtime | 运行时上下文 | 只读 |

#### 1.2 AgentState

整个 Agent 生命周期的"全局状态容器"，贯穿整个 Agent 执行流程，跨多次模型调用持久化。

| **字段** | **说明** |
|---------|---------|
| messages | 完整的对话历史 |
| user_info | 用户信息（由中间件注入） |
| user_permissions | 用户权限（由中间件注入） |

#### 1.3 request.override() 方法

用于创建修改后的请求副本：

```python
request = request.override(model=new_model)
request = request.override(messages=new_messages)
request = request.override(tools=new_tools)
```

### 2. 装饰器 dynamic_prompt 动态提示词

`@dynamic_prompt` 装饰器简化了 `wrap_model_call` 的写法，自动处理 ModelRequest/Response 的封装。

```python
@dynamic_prompt
def role_based_prompt(request: ModelRequest):
    user_role = request.runtime.context.get("user_role", "user")
    if user_role == "expert":
        return "你是一个专业气象分析师..."
    elif user_role == "beginner":
        return "你是一个友善的天气助手..."
    else:
        return "你是一个简洁的天气助手..."
```

### 7. 拓展思考

#### 7.1 override() 方法的底层原理

&emsp;&empsp;`request.override()` 返回一个新对象而不是修改原对象，这是函数式编程中"不可变性"原则的体现：

**为什么要不可变？**
1. **线程安全**：多个线程可以安全地共享和修改请求的不同副本
2. **可追溯**：原始请求始终可访问，便于调试
3. **避免副作用**：修改不会意外影响其他代码

**override() 的实现模式**：
```python
class ModelRequest:
    def __init__(self, model, messages, tools, state, runtime):
        self._model = model
        self._messages = messages
        self._tools = tools
        self._state = state
        self._runtime = runtime

    def override(self, model=None, messages=None, tools=None):
        """返回修改后的副本，保留未指定的字段"""
        return ModelRequest(
            model=model if model is not None else self._model,
            messages=messages if messages is not None else self._messages,
            tools=tools if tools is not None else self._tools,
            state=self._state,  # state 是只读的，不允许 override
            runtime=self._runtime  # runtime 是只读的，不允许 override
        )
```

#### 7.2 动态提示词的工程化实践

&emsp;&empsp;`@dynamic_prompt` 装饰器看似简单，但在生产环境中需要考虑更多细节：

**动态提示词的使用场景**：

| 场景 | 触发条件 | 提示词调整 |
|------|---------|-----------|
| 用户角色 | user_role | 专业版/简化版 |
| 对话长度 | message_count | 简短/详细 |
| 任务类型 | intent | 客服/技术/销售 |
| 时间上下文 | time_of_day | 早安/午安/晚安 |

**避免"提示词注入"**：
```python
# ❌ 危险：直接拼接用户输入到提示词
@dynamic_prompt
def dangerous_prompt(request: ModelRequest):
    user_input = request.messages[-1].content
    return f"用户说：{user_input}，请分析..."

# ✅ 安全：对用户输入进行转义/过滤
@dynamic_prompt
def safe_prompt(request: ModelRequest):
    user_input = sanitize_input(request.messages[-1].content)
    return f"用户说：{user_input}，请分析..."
```

#### 7.3 模型动态切换的成本优化

&emsp;&empsp;智能模型切换不仅是技术问题，更是成本优化问题：

**模型选择的成本公式**：
```
总成本 = Σ(模型单价 × 使用 Token 数)
```

**分层模型策略**：
| 层级 | 模型 | 单价（相对）| 适用场景 |
|------|------|------------|---------|
| L0 | GPT-4o | 1x | 复杂推理、创意任务 |
| L1 | GPT-4o-mini | 0.2x | 普通对话、简单任务 |
| L2 | GPT-3.5-turbo | 0.05x | 格式化输出、简单问答 |

**自动切换的触发条件**：
```python
def should_upgrade_model(messages, conversation_turns):
    # 条件1：对话轮数过多，切换到有更长上下文的模型
    if conversation_turns > 10:
        return True, "gpt-4o"

    # 条件2：检测到复杂推理关键词
    complex_keywords = ["分析", "比较", "评估", "设计"]
    if any(kw in messages[-1].content for kw in complex_keywords):
        return True, "gpt-4o"

    # 条件3：代码相关任务
    if "```" in messages[-1].content:
        return True, "gpt-4o"

    return False, None
```

#### 7.4 会话持久化的技术选型

&emsp;&empsp;不同的持久化方案有不同的特点和适用场景：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| InMemorySaver | 速度快、实现简单 | 服务重启丢失 | 开发测试 |
| RedisSaver | 持久化、支持分布式 | 需要额外部署 | 生产环境 |
| FileSystemSaver | 简单、无依赖 | 性能差、不支持并发 | 小规模项目 |
| 数据库Saver | 查询能力强 | 实现复杂 | 企业级应用 |

**生产环境建议**：
```python
# 开发环境：使用内存
checkpointer = InMemorySaver()

# 生产环境：使用 Redis
from langgraph.checkpoint.redis import RedisSaver

checkpointer = RedisSaver.from_conn_string(
    "redis://localhost:6379",
    session_ttl=60 * 60 * 24 * 7  # 7 天过期
)
```

#### 7.5 中间件组合的"防呆设计"

&emsp;&empsp;在设计多中间件组合时，需要防止一些常见的错误：

**常见错误**：

1. **顺序错误导致的安全漏洞**：
```python
# ❌ 错误：先执行业务后认证
middlewares = [BusinessMiddleware(), AuthMiddleware()]

# ✅ 正确：先认证后执行业务
middlewares = [AuthMiddleware(), BusinessMiddleware()]
```

2. **重复执行导致性能问题**：
```python
# ❌ 错误：同一个 Hook 点注册多个同类中间件
middlewares = [ContextCompression(), AnotherCompression()]  # 可能重复压缩

# ✅ 正确：只注册一个
middlewares = [ContextCompression()]
```

3. **状态污染**：
```python
# ❌ 错误：中间件直接修改传入的状态
def bad_middleware(state, runtime):
    state["messages"] = []  # 可能影响其他中间件

# ✅ 正确：通过返回值修改
def good_middleware(state, runtime):
    return {"truncated": True}  # 返回要合并的字段
```
