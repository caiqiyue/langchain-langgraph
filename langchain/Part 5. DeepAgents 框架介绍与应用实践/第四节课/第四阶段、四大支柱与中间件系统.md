## <center>第四阶段、四大支柱与中间件系统

### 1. 四大支柱概述

#### 1.1 四大核心内置工具

| 支柱 | 功能 | 说明 |
|------|------|------|
| 系统提示词 (System Prompts) | 定义 Agent 行为准则 | 内置 Base Agent Prompt |
| 规划工具 (Planning System) | TodoListMiddleware | write_todos 工具 |
| 文件系统 (File System) | FilesystemMiddleware | 7 个核心文件操作工具 |
| 子代理 (Sub Agents) | SubAgentMiddleware | 任务隔离与并行执行 |

#### 1.2 支柱详解

```
┌──────────────────────────────────────────────────────────────┐
│                    DeepAgents 四大支柱                         │
├────────────────┬────────────────┬────────────────┬────────────┤
│  系统提示词     │   规划工具      │   文件系统      │   子代理    │
│ System Prompt  │ Planning Tool  │ File System    │ Sub Agents │
├────────────────┼────────────────┼────────────────┼────────────┤
│ 行为总导演     │ 任务架构师      │ 上下文仓库      │ 执行特派员  │
│ 定义世界观     │ 分解任务        │ 信息持久化      │ 隔离执行    │
│ 确保协同不偏离  │ 动态调整        │ 防止溢出        │ 专业分工    │
└────────────────┴────────────────┴────────────────┴────────────┘
```

### 2. 系统提示词

#### 2.1 系统提示词的作用

系统提示词是 DeepAgents 三大中间件协同的"契约"：

**默认版本包含**：
1. **规划指令**
   - 要求 LLM 在任务开始前调用 write_todos
   - 输出格式为 JSON Schema

2. **文件操作规范**
   - write_file 用于新内容
   - edit_file 用于局部修改
   - 避免覆盖冲突

3. **子代理调用协议**
   - task 工具的参数结构
   - 结果通过 /subagent_results/\<name\>.md 回传

4. **安全底线**
   - 禁止直接执行删除、格式化等危险命令
   - 必须通过 execute 沙箱

#### 2.2 提示词组成结构

```
┌─────────────────────────────────────────────────────────┐
│              系统提示词 (System Prompt)                   │
├─────────────────────────────────────────────────────────┤
│  用户自定义部分 (User system_prompt)                      │
│    - 角色定义                                           │
│    - 任务描述                                           │
│    - 特殊指令                                           │
├─────────────────────────────────────────────────────────┤
│  内置 BASE_AGENT_PROMPT                                 │
│    - 规划指令 (write_todos 调用规范)                    │
│    - 文件操作规范 (read/write/edit 规则)               │
│    - 子代理协议 (task 工具使用规范)                     │
│    - 安全底线 (禁止危险操作)                           │
└─────────────────────────────────────────────────────────┘
```

### 3. 规划工具 (TodoList)

#### 3.1 规划工具概述

| 组件 | 工具 | 功能 |
|------|------|------|
| TodoListMiddleware | write_todos | 生成 Markdown 格式的待办事项列表 |

**功能**：
- Agent 在行动前生成 Markdown 格式的待办事项列表
- 在执行过程中更新状态（完成/进行中）
- 强制模型进行"思维链"的显性化管理

**触发条件**：
- 简单短期任务不会触发
- 复杂任务自动触发

#### 3.2 工作流程

```
┌─────────────────────────────────────────────────────────┐
│                     规划工具工作流                        │
├─────────────────────────────────────────────────────────┤
│   1. 接收复杂任务                                        │
│          ↓                                              │
│   2. write_todos 分解为子步骤 (Pending)                │
│          ↓                                              │
│   3. 执行第一步                                         │
│          ↓                                              │
│   4. 更新状态为 (Completed)                             │
│          ↓                                              │
│   5. 继续执行下一步                                     │
│          ↓                                              │
│   6. 所有步骤完成 → 生成回答                            │
└─────────────────────────────────────────────────────────┘
```

#### 3.3 Todo 状态字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | str | 唯一标识 |
| description | str | 任务描述 |
| status | str | pending/completed |
| priority | int | 优先级（可选） |
| dependencies | list[str] | 依赖关系（可选） |

### 4. 子代理 (Sub-Agent)

#### 4.1 子代理概述

| 组件 | 工具 | 说明 |
|------|------|------|
| SubAgentMiddleware | task | 任务隔离与并行执行 |

**核心概念**：
1. **任务隔离**：每个子代理有独立的上下文窗口
2. **并行执行**：支持同时启动多个子代理
3. **结果聚合**：智能整合多个子代理的输出

**使用场景**：
1. 高风险操作隔离（网页抓取、代码执行）
2. 专业化分工（主 Agent 编排，子 Agent 执行）
3. 资源优化（子 Agent 可使用轻量化模型）

#### 4.2 执行流程

```
┌─────────────────────────────────────────────────────────┐
│                     子代理执行流程                        │
├─────────────────────────────────────────────────────────┤
│   主 Agent                                              │
│      │                                                  │
│      ├──→ 子 Agent 1（独立上下文）→ 结果1              │
│      │                                                  │
│      ├──→ 子 Agent 2（独立上下文）→ 结果2              │
│      │                                                  │
│      └──→ ...                                          │
│              ↓                                         │
│         结果聚合 → 最终回答                             │
└─────────────────────────────────────────────────────────┘
```

#### 4.3 配置子代理

```python

        "name": "DocsResearcher",

        "system_prompt": "你是技术文档专家...",

    {
        "name": "CommunityResearcher",
        "description": "负责搜索社区博客的专家",
        "system_prompt": "你是社区动态专家...",
        "tools": [tavily],
        "model": "deepseek-chat"
    }
]

    model=llm,
    tools=[],

```

#### 4.4 自动触发

不传入 subagents 参数时：
- 默认会启用 SubAgentMiddleware(general_purpose_agent=True)
- Agent 自动获得 'task' 工具
- 可以调用 'general-purpose' 子 Agent

**注意**：
- 子代理有独立的 messages 和 files 命名空间
- 异常不会污染父 Agent 状态
- 支持 max_iterations 限制防止无限递归

### 5. 拓展思考

#### 5.1 四大支柱的协同关系

&emsp;&emsp;DeepAgents 的四大支柱不是孤立存在的，它们之间有紧密的协同关系：

**支柱协同图**：
```
系统提示词（定义行为边界）
        │
        ├──→ 规划工具（分解任务）
        │           │
        │           └──→ 文件系统（存储计划/结果）
        │
        └──→ 子代理（执行具体任务）
                    │
                    └──→ 四大支柱（循环）
```

**实际执行流程**：
1. **系统提示词** 定义 Agent 的角色和基本行为准则
2. **规划工具** 将复杂任务分解为可执行的 TodoList
3. **文件系统** 保存中间结果、检索历史、生成报告
4. **子代理** 在独立上下文中执行特定任务，结果返回父 Agent

#### 5.2 子代理的通信协议

&emsp;&emsp;子代理与父 Agent 之间的通信是通过"约定格式"实现的：

**结果回传格式**：
```markdown
# /subagent_results/{name}.md

## 任务概述
[子代理名称] 完成了 [任务描述]

## 执行结果
[详细结果]

## 关键发现
1. [发现1]
2. [发现2]

## 建议
[后续行动建议]
```

**命名空间隔离**：
```python
# 父 Agent 的命名空间
/subagent_results/
  ├── researcher_1.md
  ├── researcher_2.md
  └── coder.md

# 子代理 A 的命名空间（隔离）
/subagent_results/
  └── (只看到自己的结果)

# 子代理 B 的命名空间（隔离）
/subagent_results/
  └── (只看到自己的结果)
```

#### 5.3 规划工具的状态机

&emsp;&emsp;TodoListMiddleware 本质上是一个状态机，管理任务的完整生命周期：

**任务状态流转**：
```
pending → in_progress → completed
   ↑           │
   │           ↓
   └───── failed ←←←←←←←←←
```

**状态转换触发**：
```python
class TodoState:
    def transition(self, action: str) -> "TodoState":
        transitions = {
            ("pending", "start"): "in_progress",
            ("in_progress", "complete"): "completed",
            ("in_progress", "fail"): "failed",
            ("failed", "retry"): "pending",
        }
        return transitions.get((self.current, action), self.current)
```

**与 LangGraph 状态管理的对比**：
| 特性 | TodoList | LangGraph State |
|------|----------|----------------|
| 状态存储 | Memory / 文件 | Checkpointer |
| 持久化 | 可选 | 必须 |
| 状态共享 | 通过文件 | 原生支持 |
| 适用场景 | 任务跟踪 | 复杂工作流 |

#### 5.4 中间件与支柱的关系

&emsp;&emsp;DeepAgents 的中间件系统与四大支柱有重叠和补充：

**Middleware vs 支柱对比**：

| 支柱 | 层级 | Middleware 对应 |
|------|------|----------------|
| 系统提示词 | 行为定义 | PromptMiddleware |
| 规划工具 | 任务管理 | TodoListMiddleware |
| 文件系统 | 资源访问 | FilesystemMiddleware |
| 子代理 | 任务执行 | SubAgentMiddleware |

**自定义中间件可以增强支柱**：
```python
# 增强规划工具：添加优先级管理
class PriorityTodoMiddleware(AgentMiddleware):
    def before_agent(self, state, runtime):
        todos = extract_todos(state)
        # 按优先级排序
        todos.sort(key=lambda t: t.priority, reverse=True)
        return {"todos": todos}

# 增强文件系统：添加访问审计
class AuditFilesystemMiddleware(AgentMiddleware):
    def before_tool_call(self, tool_name, args):
        if tool_name in ["read_file", "write_file", "execute"]:
            audit_log.record(
                action=tool_name,
                path=args.get("path"),
                timestamp=time.time()
            )
```
