## <center>第五阶段、文件系统与沙箱系统

### 1. 文件系统工具

#### 1.1 工具总览

| 工具 | 功能 | 特点 |
|------|------|------|
| ls | 浏览目录结构 | 支持模式匹配 |
| read_file | 读取文件内容 | 支持 offset/limit 分页 |
| write_file | 创建新文件 | 整体覆盖写入 |
| edit_file | 修改文件 | 支持精确字符串替换 |
| glob | 通配符查找文件 | 如 *.py, **/*.md |
| grep | 正则表达式搜索 | 代码定位神器 |
| execute | 执行 Shell 命令 | 需 Sandbox 支持 |

#### 1.2 read_file 分页机制

**分页参数**：
- offset：偏移量（行数）
- limit：每次读取的行数（默认 500）

**作用**：
- 强制 Agent 对大文件进行分页读取
- 避免一次性读爆上下文
- 适合处理大型日志文件

**示例**：
```
第一次：read_file("/large_log.txt", offset=0, limit=500)
第二次：read_file("/large_log.txt", offset=500, limit=500)
...
```

#### 1.3 edit_file 精确替换

**参数**：
- path：文件路径
- old_string：要被替换的字符串
- new_string：替换后的字符串
- replace_all：是否全部替换（默认 False）

**防呆设计**：
- 要求先读后改
- 避免误操作覆盖整个文件

**示例**：
```python
edit_file(
    path="/config.py",
    old_string="DEBUG = True",
    new_string="DEBUG = False"
)
```

#### 1.4 大结果自动转存

**机制**：
当工具调用产生的结果过长时：
1. 系统自动拦截
2. 将完整结果写入文件系统（路径如：/large_tool_results/{id}）
3. 只给 Agent 返回摘要和文件路径

**解决的问题**：
- 搜索结果或日志文件过长导致 Agent 崩溃
- 上下文溢出

### 2. Backend后端系统

#### 2.1 Backend 类型对比

| Backend | 存储位置 | 执行环境 | 适用场景 |
|---------|---------|---------|----------|
| FilesystemBackend | 本地磁盘 | 宿主机 | 本地开发、快速原型 |
| DockerBackend | 本地磁盘 | Docker 容器 | 安全隔离、标准化环境 |
| E2BBackend | 云端 | 云端沙箱 | 云端部署、多租户 |
| StoreBackend | 数据库 | 宿主机 | 数据持久化、查询 |
| CompositeBackend | 混合 | 混合 | 性能优化、混合云 |

#### 2.2 FilesystemBackend

```python
backend = FilesystemBackend(
    root_dir="./workspace",
    virtual_mode=True  # 启用虚拟文件系统
)
```

**特点**：
- 简单易用
- 适合本地开发
- virtual_mode 将 Agent 操作映射到指定目录

#### 2.3 DockerBackend

```python
backend = DockerBackend(
    image="python:3.11-slim",
    auto_remove=True,
    cpu_quota=50000,
    memory_limit="512m",
    network_disabled=False
)
```

**特点**：
- 安全隔离（代码在容器内运行）
- 环境一致性（标准化镜像）
- 生命周期自动管理
- 可配置资源限制

#### 2.4 E2BBackend

```python
backend = E2BBackend(
    template="base",
    api_key=os.getenv("E2B_API_KEY")
)
```

**特点**：
- 无需本地 Docker
- 云端安全隔离
- 适合生产环境
- 按使用时长计费

#### 2.5 Backend vs Checkpointer vs Store

| 组件 | 层级 | 负责 | 生命周期 |
|------|------|------|----------|
| Backend | 环境层 | 文件存在哪、代码在哪跑 | 任务级 |
| Checkpointer | 状态层 | 当前对话上下文 | 线程级 |
| Store | 记忆层 | 跨会话知识积累 | 全局级 |

**比喻**：
- Backend = 工作台/电脑
- Checkpointer = 大脑工作记忆
- Store = 日记本/知识库

### 3. interrupt_on配置

#### 3.1 interrupt_on 概述

interrupt_on 是 HITL（Human-in-the-Loop）开关：

- 类型：`dict[str, bool | InterruptOnConfig]`
- 映射"工具名称"到"中断配置"
- 当 Agent 调用指定工具时暂停执行
- 等待人工审批后才能继续

**典型使用**：
```python
interrupt_on={"write_file": True}
# 当 Agent 试图调用 write_file 时中断
```

#### 3.2 配置示例

```python
from langgraph.checkpoint.memory import InMemorySaver
from langchain_tavily import TavilySearch

# 创建 Agent，配置中断
agent = create_deep_agent(
    model=llm,
    tools=[TavilySearch(max_results=2)],
    backend=FilesystemBackend(root_dir="./workspace"),
    checkpointer=InMemorySaver(),  # 必须！支持中断和恢复
    interrupt_on={
        "write_file": True,     # 写入文件时中断
        "tavily_search": True, # 搜索时中断
    }
)
```

#### 3.3 审批操作类型

| 操作 | 说明 |
|------|------|
| ApproveDecision | 批准操作继续执行 |
| RejectDecision | 拒绝操作，停止执行 |
| EditDecision | 编辑参数后继续执行 |

#### 3.4 完整流程

```
┌─────────────────────────────────────────────────────────┐
│                    HITL 完整流程                          │
├─────────────────────────────────────────────────────────┤
│   1. Agent.invoke() 执行                               │
│          ↓                                              │
│   2. 遇到配置了 interrupt_on 的工具调用                   │
│          ↓                                              │
│   3. 执行暂停（Suspend）                                │
│          ↓                                              │
│   4. 人工介入审查                                        │
│          ↓                                              │
│   5a. 批准 → 继续执行                                   │
│   5b. 拒绝 → 停止                                       │
│   5c. 编辑 → 修改参数后继续                             │
└─────────────────────────────────────────────────────────┘
```

### 4. 拓展思考

#### 4.1 沙箱系统的安全边界

&emsp;&emsp;Docker 沙箱是 Agentic RAG 的重要基础设施，理解其安全边界至关重要：

**Docker 隔离的安全层级**：

| 层级 | 保护内容 | 潜在风险 |
|------|---------|---------|
| 进程隔离 | 容器内进程不访问宿主机 | 内核漏洞逃逸 |
| 文件系统隔离 | 除非挂载，容器内无法访问宿主机文件 | 卷挂载配置错误 |
| 网络隔离 | 默认容器网络与宿主机隔离 | 网络模式配置 |
| 资源限制 | CPU/内存配额 | DoS 攻击 |

**危险操作示例**：
```python
# 在沙箱中执行这些命令是安全的
result = backend.execute("rm -rf /")  # 只删除容器内文件

# 但如果挂载了卷，就可能影响宿主机
# backend = DockerBackend(volumes={"/host/path": "/container/path"})
# result = backend.execute("rm -rf /host/path")  # 危险！
```

**安全最佳实践**：
1. 永远不要在生产环境挂载敏感目录
2. 使用 `read_only` 文件系统
3. 禁用特权容器
4. 限制网络访问

#### 4.2 FilesystemBackend vs DockerBackend

&emsp;&emsp;两种 Backend 有不同的适用场景：

| 场景 | FilesystemBackend | DockerBackend |
|------|------------------|---------------|
| 开发调试 | ✅ 快速迭代 | ❌ 启动慢 |
| 代码执行安全 | ❌ 无隔离 | ✅ 沙箱隔离 |
| 持久化需求 | ✅ 直接操作文件 | ❌ 需要挂载 |
| 环境一致性 | ❌ 依赖宿主机 | ✅ 标准化镜像 |

**混合使用策略**：
```python
# 开发环境：FilesystemBackend
if settings.ENV == "development":
    backend = FilesystemBackend(root_dir="./workspace")

# 生产环境：DockerBackend
else:
    backend = DockerBackend(
        image="python:3.11-slim",
        auto_remove=True,
        network_disabled=False  # 如果需要网络访问
    )
```

#### 4.3 interrupt_on 的深度应用

&emsp;&emsp;HITL 不仅仅是"人工审批"，还可以用于更复杂的场景：

**智能断点**：
```python
# 根据操作风险等级动态中断
def calculate_risk(tool_name, args):
    if tool_name == "write_file":
        # 写入操作根据文件类型判断风险
        if args.get("path", "").endswith(".py"):
            return "medium"  # Python 文件中等风险
        elif args.get("path", "").endswith(".env"):
            return "high"  # 环境变量高风险
    elif tool_name == "execute":
        return "critical"  # 执行命令最高风险
    return "low"

# 配置动态中断
interrupt_on = {
    "write_file": True,
    "execute": True,  # 所有执行都中断
}
```

**审批后的自动分类**：
```python
# 根据审批决策更新 Agent 的"知识库"
if decision == ApproveDecision:
    # 批准的操作可以作为示例被记住
    memory.remember_action(tool_name, args, approved=True)
elif decision == RejectDecision:
    # 拒绝的操作被记录为"边界"
    memory.remember_boundary(tool_name, args)
```

#### 4.4 大结果自动转存的底层机制

&emsp;&emsp;当工具调用产生的结果过长时，DeepAgents 会自动将结果转存到文件系统：

**触发条件**：
- 搜索结果超过 10,000 字符
- 日志文件读取超过 5,000 行
- 任何工具返回超过 Token 限制

**转存路径格式**：
```
/large_tool_results/{tool_name}_{timestamp}_{uuid}.txt
```

**对 Agent 的影响**：
- Agent 不会直接看到完整结果
- 只能看到摘要和文件路径
- 需要主动调用 `read_file` 读取完整内容

**设计目的**：
1. 防止上下文溢出
2. 强制 Agent 进行"有意识的"长文本处理
3. 便于审计和复查
