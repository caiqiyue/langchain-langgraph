## <center>第七阶段、复杂业务编排</center>

&emsp;&emsp;当单个 Agent 无法满足复杂业务需求时，需要构建多 Agent 协作系统或使用条件分支、循环等编排模式。

### 1. 多 Agent 协作架构

#### 层次化 Agent 架构

```
┌─────────────────────────────────────────────────────┐
│              Router Agent (路由)                     │
│   负责：理解用户意图，分发到子 Agent                  │
└─────────────────────┬───────────────────────────────┘
                      │
        ┌─────────────┼─────────────┬─────────────┐
        ↓             ↓             ↓             ↓
   [搜索Agent]  [计算Agent]  [天气Agent]  [邮件Agent]
   专门搜索     专门计算    专门查天气   专门发邮件
```

#### 多 Agent 协作优势

1. **专业化分工**：每个子 Agent 专注于特定领域，工具描述更清晰
2. **降低复杂度**：单个 Agent 工具数量减少，选错工具概率降低
3. **可扩展性**：新增能力只需添加子 Agent

### 2. 条件分支流程

基于 Agent 输出决定执行路径，适用于审批流程等场景。

```python
# 根据 Agent 输出决定分支
if "approved" in response:
    # 执行审批通过流程
    pass
elif "rejected" in response:
    # 执行审批拒绝流程
    pass
else:
    # 需要更多信息
    pass
```

### 3. 循环与重试机制

重复执行直到满足条件，适用于数据处理等场景。

```python
config = {
    "configurable": {
        "thread_id": "retry_demo",
        "recursion_limit": 5  # 最多重试5次
    }
}

# 错误处理
try:
    result = agent.invoke({"messages": [...]}, config=config)
except Exception as e:
    # 重试逻辑
    pass
```

### 6. 拓展思考

#### 6.1 多 Agent 架构的深层挑战

&emsp;&emsp;多 Agent 协作虽然强大，但也带来了新的挑战：

**通信开销**：Agent 之间的消息传递会增加延迟和 Token 消耗。在设计时应避免不必要的跨 Agent 调用。

**状态一致性**：多个 Agent 共享状态时，需要考虑并发修改的问题。LangGraph 的 Checkpoint 机制可以帮助维护状态一致性。

**调试复杂性**：多 Agent 系统的 bug 往往更难复现，因为涉及多个模型的交互。建议使用 LangSmith 进行全链路追踪。

#### 6.2 路由 Agent 的设计模式

&emsp;&emsp;路由 Agent（Router Agent） 是多 Agent 系统中的关键组件，其设计有几种常见模式：

**模式一：意图分类**
```python
# 根据用户意图直接路由到子 Agent
intents = ["搜索", "计算", "天气", "邮件"]
routing = {intent: agent for intent, agent in zip(intents, [search_agent, calc_agent, weather_agent, email_agent])}
```

**模式二：LLM 动态路由**
```python
# 让 LLM 决定使用哪个 Agent
system_prompt = """分析用户问题，选择最合适的 Agent：
- search_agent：搜索信息
- calc_agent：数学计算
- weather_agent：天气查询
只返回 Agent 名称，不要其他内容。"""
```

**模式三：多层路由**
```
用户 → 路由层1（领域） → 路由层2（任务类型） → 具体 Agent
```

#### 6.3 递归限制的深层理解

&emsp;&emsp;`recursion_limit` 是防止 Agent 进入无限循环的关键参数，但其行为比表面看起来更复杂：

**实际限制的对象**：
- 限制的是 **Agent 推理循环的次数**，不是工具调用的总次数
- 一个推理循环可能包含多次工具调用

**建议的配置策略**：
| 场景 | recursion_limit | 说明 |
|------|----------------|------|
| 简单问答 | 5-10 | 1-2 次工具调用 |
| 复杂研究 | 20-30 | 需要多次检索 |
| 代码生成 | 15-25 | 编译-修正循环 |
| 长任务 | 50+ | 配合 interrupt_on |

**最佳实践**：
1. 从较小的值开始（如 10），根据实际需求调大
2. 结合 `interrupt_on` 对高风险操作进行人工审核
3. 使用 LangSmith 分析递归限制触发的真实原因

#### 6.4 错误处理与容错设计

&emsp;&emsp;在生产环境中，Agent 编排必须考虑各种异常情况：

**错误分类与处理策略**：

| 错误类型 | 可能原因 | 处理策略 |
|---------|---------|---------|
| 工具超时 | 网络问题、工具卡死 | 设置超时、重试机制 |
| 工具返回错误 | 参数错误、权限问题 | 记录日志、返回友好提示 |
| LLM 生成错误 | API 限流、模型故障 | 降级处理、切换模型 |
| 递归超限 | 任务复杂度过高 | 拆解任务、人工介入 |

**容错架构示例**：
```python
from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()

try:
    result = agent.invoke(
        {"messages": [...]},
        config={"configurable": {"checkpointer": checkpointer}}
    )
except TimeoutError:
    # 重试逻辑
    result = retry_with_timeout(agent, max_retries=3)
except Exception as e:
    # 降级处理
    logger.error(f"Agent 执行失败: {e}")
    result = fallback_response()
```
